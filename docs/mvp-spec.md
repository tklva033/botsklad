# MVP Specification

## 1. Цель MVP

Запустить Telegram-бота, который закрывает ежедневные складские операции без отдельного мобильного приложения:

- найти товар;
- принять товар;
- выдать товар;
- переместить товар;
- проверить остатки;
- провести ревизию;
- отследить, кто и когда изменил остаток.

## 3. Что должен уметь бот

## Авторизация

- вход по номеру телефона;
- привязка Telegram-аккаунта к сотруднику;
- проверка роли перед каждым действием.

## Поиск товара

Вход:

- название;
- часть названия;
- артикул;
- QR-код.

Выход:

- название;
- артикул;
- остаток;
- местоположение;
- последнее поступление;
- фото, если есть.

## Приемка

Сценарий:

1. Пользователь выбирает `📥 Приемка`.
2. Бот спрашивает товар.
3. Бот спрашивает количество.
4. Бот спрашивает локацию.
5. Бот показывает подтверждение.
6. После подтверждения создает операцию и обновляет остаток.

## Выдача

Сценарий:

1. Пользователь выбирает `📤 Выдача`.
2. Бот спрашивает товар.
3. Бот показывает доступный остаток.
4. Бот спрашивает количество.
5. Бот спрашивает, кому выдали.
6. Бот подтверждает списание и сохраняет операцию.

Правило:

- нельзя выдать больше доступного остатка без отдельного права администратора.

## Перемещение

Сценарий:

1. Выбрать товар.
2. Указать откуда.
3. Указать куда.
4. Указать количество.
5. Сохранить операцию перемещения.

## Остатки

Показать:

- общий остаток товара;
- остаток по ячейкам;
- товары ниже минимального уровня.

## Ревизия

Сценарий:

1. Пользователь выбирает зону или стеллаж.
2. Бот по очереди показывает товары по локациям.
3. Кладовщик вводит фактическое количество.
4. При расхождении фиксируется запись ревизии.
5. По завершении формируется список несоответствий.

## История товара

Показать последние операции:

- дата;
- тип операции;
- количество;
- сотрудник;
- комментарий.

## Уведомления

Минимум для MVP:

- товар ниже минимального остатка;
- завершена приемка;
- зафиксировано расхождение по ревизии.

## 4. Приоритет бизнес-логики

### Критично

- каждая операция должна быть атомарной;
- нельзя допускать отрицательный остаток без специального права;
- каждая смена остатка должна оставлять след в журнале;
- роль пользователя должна проверяться на сервере, а не только в меню.

### Желательно

- защита от дублей при повторной отправке сообщений;
- понятные тексты ошибок;
- быстрый поиск с подсказками.

## 6. Предлагаемый стек

Если делать систему как масштабируемый продукт:

- `Node.js + TypeScript`
- `Telegraf` для Telegram-бота
- `PostgreSQL` как основная база
- `Prisma` как ORM
- `Redis` для сессий, очередей и антидублей
- `BullMQ` для фоновых задач и уведомлений
- `S3-совместимое хранилище` для фото товаров и резервных файлов

Если нужен быстрый внутренний запуск:

- `Node.js + TypeScript`
- `Telegraf`
- `PostgreSQL`
- без Redis на первом этапе

## 8. Что не стоит делать в первой версии

Лучше не перегружать стартовую версию такими функциями:

- поиск по фотографии;
- распознавание речи;
- сложная карта склада;
- глубокая BI-аналитика;
- многошаговые бизнес-процессы согласования.

Они полезны, но не мешают запустить сильный базовый продукт.

## 9. Как выглядит хороший план запуска

### Этап 1. База

- пользователи и роли;
- товары, локации и остатки;
- операции приемки, выдачи и перемещения.

### Этап 2. Контроль

- история операций;
- ревизия;
- расхождения;
- уведомления о минимальном остатке.

### Этап 3. Управление

- отчеты;
- импорт и экспорт;
- заявки и согласования.

## 10. Формулировка продукта

`Bot Sklad` это Telegram-система складского учета для команд, которым нужен быстрый доступ к остаткам, операциям и ревизии без внедрения тяжелой WMS.
